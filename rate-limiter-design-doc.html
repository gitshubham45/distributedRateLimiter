Distributed Rate Limiter Design Document

1. Project Overview

This project is a Go/Gin based rate limiter service. It registers endpoint-specific rate-limit configuration, checks incoming requests, and forwards allowed requests to backend services using a reverse proxy.

The current implementation is local and in-memory. It is structured so database-backed endpoint registration and Redis-backed distributed counters can be added later.

2. Current Features

  • Register endpoint-specific rate-limit rules.
  • Select limiter strategy by name.
  • Current strategies: per_unit_time and placeholder leaky_bucket.
  • Track per-unit-time request windows in memory.
  • Load backend service targets from services.yaml.
  • Forward allowed requests using Go's reverse proxy.
  • Log request and application events using Zap.

3. API Endpoints

3.1 Health Check

GET /health

Returns:

{
  "message": "OK"
}

3.2 Register Endpoint

POST /register-endpoint

Example request:

{
  "path": "/ww",
  "method": "GET",
  "target_service": "user",
  "strategy": {
    "name": "per_unit_time",
    "limit": 10,
    "interval": 60
  }
}
Field Description
path Public path handled by the rate limiter.
method HTTP method used to build the registry key.
target_service Service name from services.yaml.
strategy.name Limiter strategy name, for example per_unit_time.
strategy.limit Maximum requests allowed in the interval.
strategy.interval Window size in seconds.

3.3 Call Registered Endpoint

GET /ww?client_id=test-user

The current per-unit-time limiter reads client_id from the query string. If the endpoint is registered and the request is allowed, the request is forwarded to the configured backend service.

4. Runtime Flow

Client request
  ↓
Gin route / NoRoute
  ↓
HandleLimit
  ↓
Lookup "METHOD:/path" in in-memory registry
  ↓
Call limiter strategy Allow(...)
  ↓
If allowed, resolve target_service from services.yaml
  ↓
Forward request to backend service

5. Data Model: Endpoint Registration

EndpointRegisterReq
  ├── path
  ├── method
  ├── target_service
  └── StrategyConfig
        ├── name
        ├── limit
        └── interval

EndpointRegisterReq
  └── creates LimitConfig
        ├── path
        ├── target_service
        ├── limit_strategy_name
        ├── limiter
        └── config

LimitRegistry
  └── key: "METHOD:/path"
      value: LimitConfig

Current behavior:

  • POST /register-endpoint accepts endpoint registration input.
  • The registry key is built as method + ":" + path.
  • The registered config is stored in memory in Limiter.LimitRegistery.
  • The database write is not implemented yet.

6. Data Model: Strategy Registry

StrategyRegistry
  ├── "per_unit_time" -> PerUnitTimeLimit
  └── "leaky_bucket"  -> LeakyBucket

RequestLimiter interface
  └── Allow(c, config) bool

PerUnitTimeLimit implements RequestLimiter
  ├── requestWindow map
  └── mutex

LeakyBucket implements RequestLimiter
  └── currently placeholder

limiter.GetRequestLimiter(name) checks the strategy registry. If the strategy object already exists, it returns the same object. Otherwise it creates and stores it.

7. Data Model: Runtime Rate Limiting

HTTP Request
  ├── method
  ├── path
  └── query param: client_id
          │
          ▼
Registry lookup by "METHOD:/path"
          │
          ▼
LimitConfig
  └── StrategyConfig
        ├── name
        ├── limit
        └── interval
          │
          ▼
PerUnitTimeLimit
  └── requestWindow["METHOD:/path:client_id"]
        ├── last_reset_at
        └── count

Current per-unit-time algorithm:

  1. Read client_id from the query string.
  2. Build a request-window key from method, path, and client ID.
  3. If no window exists, create one with count 1.
  4. If the configured interval has expired, reset count to 1.
  5. If count is below the configured limit, increment and allow.
  6. Otherwise reject with HTTP 429.

10. Future DB Model

SERVICES
  ├── name PK
  └── target_url

ENDPOINTS
  ├── id PK
  ├── method
  ├── path
  ├── target_service FK -> SERVICES.name
  ├── strategy_name
  ├── limit
  ├── interval
  ├── created_at
  └── updated_at

The database should be the source of truth for endpoint configuration. The application should load endpoint config into memory to avoid fetching from the DB on every request.

11. Future Redis Model

HTTP Request
  └── increments Redis key

Redis key
  ├── key: rate_limit:{METHOD}:{path}:{client_id}:{window_start}
  ├── count
  └── ttl_seconds

Redis should be used for distributed runtime counters so multiple service instances share the same rate-limit state.

12. Current Limitations

  • Endpoint registration is in memory only.
  • Rate-limit counters are in memory only.
  • Restarting the app clears registrations and counters.
  • Multiple service instances will not share counters.
  • leaky_bucket is currently a placeholder.
  • There is no DB persistence yet.
  • There is no Redis-backed distributed counter yet.

13. Recommended Next Steps

  1. Persist endpoint registrations in a database.
  2. Load endpoint configs into an in-memory cache at startup.
  3. Use Redis for distributed request counters.
  4. Add cache invalidation when endpoint config changes.
  5. Implement the leaky bucket strategy.
  6. Add tests for registration, missing endpoint, allowed request, and rejected request.
